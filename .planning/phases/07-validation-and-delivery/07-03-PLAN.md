# Architecture Design Document

## 1. System Overview

Brief description of the distributed checkout system: Order, Stock, Payment domain services coordinated by a SAGA orchestrator, backed by per-domain Redis Clusters, accessed through an nginx gateway.

Include a Mermaid `graph LR` diagram showing:
- Client → Gateway (nginx :8000)
- Gateway → Order Service (:5000)
- Gateway → Stock Service (:5000)
- Gateway → Payment Service (:5000)
- Order Service → Orchestrator (gRPC :50053)
- Orchestrator → Stock Service (gRPC :50051)
- Orchestrator → Payment Service (gRPC :50052)
- Order Service → Order Redis Cluster
- Stock Service → Stock Redis Cluster
- Payment + Orchestrator → Payment Redis Cluster (shared)
- Orchestrator → Redis Streams (event publishing)

## 2. Communication: gRPC Design

**What:** Proto definitions for StockService (ReserveStock, ReleaseStock, CheckStock) and PaymentService (ChargePayment, RefundPayment, CheckPayment). Dual-server pattern — each service runs HTTP on :5000 (for external API) and gRPC (for inter-service orchestration). Every mutation RPC includes an idempotency_key field.

**Alternatives considered:**
- HTTP/REST for inter-service calls: Simpler but no type safety, no code generation, higher latency, harder to enforce idempotency contracts
- Message queue (e.g., RabbitMQ): Asynchronous but adds infrastructure complexity and makes synchronous checkout flow harder

**Why gRPC:** Typed contracts via protobuf enforce interface correctness at compile time. The idempotency_key field is mandatory in the proto definition — impossible to forget. Low latency for synchronous SAGA steps. Code generation eliminates manual serialization.

Include a Mermaid `sequenceDiagram` showing the checkout flow:
- Order → Orchestrator: CheckoutRequest (gRPC)
- Orchestrator → Stock: ReserveStock (gRPC, idempotency_key)
- Stock → Orchestrator: ReserveStockResponse
- Orchestrator → Payment: ChargePayment (gRPC, idempotency_key)
- Payment → Orchestrator: ChargePaymentResponse
- Orchestrator → Order: CheckoutResponse

## 3. Orchestration: SAGA Pattern

**What:** A dedicated SAGA orchestrator service coordinates checkout as a state machine. States: STARTED → STOCK_RESERVED → PAYMENT_CHARGED → COMPLETED. On any failure, transitions to COMPENSATING and runs reverse compensation (refund payment → release stock → FAILED). SAGA state persisted in Redis hash per checkout (survives crashes). State transitions use a Lua CAS (compare-and-swap) script to prevent invalid jumps.

**Alternatives considered:**
- Choreography-only (no orchestrator): Each service reacts to events and drives its own compensation. Harder to make crash-recoverable — no single owner to detect stuck transactions.
- Two-Phase Commit (2PC): Redis doesn't support XA transactions. Even if it did, 2PC blocks on coordinator failure.

**Why SAGA orchestrator:** Single owner for each transaction makes crash recovery straightforward — scan for non-terminal SAGAs on startup and drive them to completion or compensation. Idempotent service operations (via Lua scripts) make replay safe. Explicit state machine prevents invalid state jumps. Compensation is guaranteed (retry with backoff until success).

Include a Mermaid `stateDiagram-v2` showing:
- [*] → STARTED
- STARTED → STOCK_RESERVED (reserve OK)
- STARTED → COMPENSATING (reserve failed)
- STOCK_RESERVED → PAYMENT_CHARGED (charge OK)
- STOCK_RESERVED → COMPENSATING (charge failed)
- PAYMENT_CHARGED → COMPLETED (confirm OK)
- COMPENSATING → FAILED (compensation done)
- COMPLETED → [*]
- FAILED → [*]

## 4. Event-Driven: Redis Streams

**What:** SAGA lifecycle events (checkout started, stock reserved, payment completed, compensation triggered) are published to a Redis Stream. Consumer groups with XREADGROUP provide at-least-once delivery. XAUTOCLAIM reclaims idle messages (Redis 6.2+). Dead-letter stream for messages exceeding max delivery attempts. Event publishing is fire-and-forget — never blocks the checkout path.

**Alternatives considered:**
- Apache Kafka: Industry standard for event streaming, but requires separate infrastructure (brokers, ZooKeeper/KRaft), significantly increases CPU/memory footprint beyond 20-CPU budget
- RabbitMQ: Mature message broker, but adds another service to manage. Redis Streams provides similar consumer group semantics natively.

**Why Redis Streams:** Same Redis infrastructure already in use — no additional services. Built-in consumer groups with acknowledgment. XAUTOCLAIM handles consumer crashes. Fits within the 20-CPU budget constraint. The fire-and-forget publishing pattern means a Redis outage drops events (logged + counted) but never blocks checkout transactions.

Include a Mermaid `sequenceDiagram` showing:
- Orchestrator → Stream: XADD event
- Consumer → Stream: XREADGROUP (block)
- Stream → Consumer: event delivered
- Consumer processes event
- Consumer → Stream: XACK
- Note: If consumer crashes, XAUTOCLAIM re-delivers

## 5. Fault Tolerance Strategy

**What:** Three layers of fault tolerance:
1. **Circuit breakers** — Independent per-service breakers (Stock, Payment) with failure_threshold=5, recovery_timeout=30s. Open breaker means service is down — no retries, immediate failure, compensation triggered.
2. **Startup SAGA recovery** — On orchestrator restart, scan Redis for non-terminal SAGAs older than a staleness threshold. Drive each to completion (forward-first using idempotent keys) or compensation. Blocks serving until recovery completes.
3. **Container self-healing** — `restart: always` on all Docker containers. Crashed containers restart automatically.

**Alternatives considered:**
- No recovery scanner (rely on timeouts): SAGAs stuck in partial state would never resolve. Violates the "no lost money" requirement.
- Distributed locking (Redlock) for coordination: Known edge cases (split-brain with clock skew). Idempotency keys + CAS transitions are simpler and correct.
- Retrying circuit-breaker-open calls: Wastes time when a service is genuinely down. Better to fail fast and compensate.

**Why this approach:** The orchestrator owns each SAGA — no coordination needed between replicas (runs as single replica). Idempotent operations make replay safe at any point. Circuit breakers contain cascade failures. The combination guarantees eventual consistency: after recovery, every checkout is either fully completed or fully compensated.

## 6. Infrastructure: Redis Cluster + Kubernetes

**What:**
- **Redis Cluster:** 3 independent clusters (Order, Stock, Payment+Orchestrator), each with 3 primary + 3 replica nodes. AOF persistence, noeviction policy. Automatic failover when a primary fails. Hash tags ({item:}, {user:}, {saga:}) ensure related keys co-locate on the same slot — required for Lua scripts that operate on multiple keys.
- **Kubernetes:** HPA (Horizontal Pod Autoscaler) scales Order, Stock, Payment replicas on CPU > 70%. Orchestrator runs as single replica (avoids split-brain for SAGA ownership). Bitnami Helm charts for Redis Cluster deployment. Total budget: 20 CPUs.

**Alternatives considered:**
- Single shared Redis instance: No fault isolation — one service's load spike affects all. No automatic failover.
- Redis Sentinel (master-replica without clustering): Provides failover but not sharding. Sufficient for this scale, but Redis Cluster provides both.
- Multiple orchestrator replicas with distributed locking: Complex and error-prone. Single replica with restart:always is simpler and correct for the scale.

**Why this approach:** Per-domain Redis Clusters provide fault isolation — a Stock Redis failure doesn't affect Payment. Automatic failover ensures no manual intervention needed. HPA handles traffic spikes within the CPU budget. Single orchestrator replica eliminates split-brain. Hash tags are the Redis Cluster way to guarantee atomicity for multi-key Lua scripts.

Include a Mermaid `graph TD` showing:
- Three Redis Cluster subgraphs (Order, Stock, Payment+Orchestrator)
- Each with 3 primary + 3 replica nodes
- Services connected to their respective clusters
```
