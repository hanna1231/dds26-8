**1. Add a `benchmark` target to the Makefile:**

Add the following target after the existing `test` target in `/Users/daniel/WebstormProjects/dds26-8/Makefile`:

```makefile
# Clone and run the wdm-project-benchmark consistency test against live cluster
benchmark:
	@if [ ! -d "wdm-project-benchmark" ]; then \
		echo "Cloning wdm-project-benchmark..."; \
		git clone https://github.com/delftdata/wdm-project-benchmark; \
	fi
	@echo '{"ORDER_URL": "http://localhost:8000", "PAYMENT_URL": "http://localhost:8000", "STOCK_URL": "http://localhost:8000"}' > wdm-project-benchmark/urls.json
	@cd wdm-project-benchmark && pip install -r requirements.txt -q 2>/dev/null || true
	@echo "Running consistency test against http://localhost:8000..."
	cd wdm-project-benchmark && python -m consistency-test.run_consistency_test
```

Also add `benchmark` to the `.PHONY` declaration at the top of the Makefile.

Add `wdm-project-benchmark/` to `.gitignore` if not already present (the benchmark repo should not be committed).

**2. Ensure the cluster is running:**

Before running the benchmark, the full Docker Compose cluster must be up. Run:
```bash
make dev-up
```

Wait for the cluster to fully initialize (the target already has a 15-second sleep). Verify services are healthy:
```bash
curl -s http://localhost:8000/health 2>/dev/null || echo "Gateway not ready"
```

If the gateway returns 502 or connection refused, wait and retry. The Bitnami Redis cluster may take up to 30 seconds to finish slot negotiation.

**3. Run the benchmark:**

```bash
make benchmark
```

The benchmark will:
1. Create 1 item with price=1 and add 100 stock
2. Create 1000 users with 1 credit each
3. Fire 1000 concurrent checkout requests
4. Assert: item stock = 0, total credits = 900 (1000 - 100 successful checkouts)

**4. If the benchmark fails — diagnose and fix:**

**Connection errors (HTTP 502, connection refused):**
- Ensure `make dev-up` or `make dev-cluster` was run and services are healthy
- Check `docker compose ps` for crashed containers
- Check `docker compose logs order-service` for Redis connection errors
- The gateway_nginx.conf proxies to service containers — verify routing

**Consistency violations (stock != 0 or credits != 900):**
- This indicates the SAGA orchestrator is not correctly coordinating checkout
- Check orchestrator logs: `docker compose logs orchestrator-service`
- Look for compensation failures, circuit breaker trips, or stuck SAGAs
- If SAGAs are stuck in non-terminal states, this is a correctness issue that must be fixed in the service code (not the benchmark)

**Benchmark script errors (ImportError, ModuleNotFoundError):**
- The benchmark's `consistency-test/` is a Python package
- If `python -m consistency-test.run_consistency_test` fails, try `cd consistency-test && python run_consistency_test.py`
- Install missing deps: `pip install aiohttp requests`

**Timeout or slow responses:**
- The benchmark uses asyncio.gather for 1000 concurrent requests
- If checkouts are slow, check service CPU usage and Redis cluster health
- This is a performance issue — triage after correctness is confirmed

**5. Capture results:**

Record the benchmark output showing:
- Number of successful/failed checkouts
- Final stock count
- Final credit sum
- Consistency verdict (PASS/FAIL)

IMPORTANT: Do NOT modify the benchmark code. Only configure urls.json and run it.
