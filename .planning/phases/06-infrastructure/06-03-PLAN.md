Create a new `Makefile` in the project root with developer-friendly targets:

```makefile
.PHONY: dev-up dev-cluster dev-down dev-logs dev-build dev-clean test dev-status

# Start simplified topology (single shared 6-node Redis cluster) — fast local dev
dev-up: dev-build
	ORDER_REDIS_HOST=shared-redis-0 \
	STOCK_REDIS_HOST=shared-redis-0 \
	PAYMENT_REDIS_HOST=shared-redis-0 \
	ORCH_REDIS_HOST=shared-redis-0 \
	docker compose --profile simple up -d
	@echo "Waiting for shared Redis cluster to initialize..."
	@sleep 15
	@echo "Simple mode: single shared 6-node Redis cluster"
	@echo "API available at http://localhost:8000"

# Start full 3-cluster topology (18 Redis nodes + 5 app services) — mirrors production
dev-cluster: dev-build
	docker compose --profile full up -d
	@echo "Waiting for Redis clusters to initialize..."
	@sleep 15
	@echo "Full mode: 3 independent Redis clusters (18 nodes)"
	@echo "API available at http://localhost:8000"

# Build all service images
dev-build:
	docker compose build

# Stop all containers
dev-down:
	docker compose down

# Follow logs for all services (Ctrl+C to stop)
dev-logs:
	docker compose logs -f order-service stock-service payment-service orchestrator-service gateway

# Stop and remove all containers, networks, and volumes (CLEAN SLATE)
dev-clean:
	docker compose down -v --remove-orphans

# Run tests (standalone Redis, not cluster)
test:
	pytest tests/ -x -v

# Show status of all containers
dev-status:
	docker compose ps
```

Key design decisions:
- `dev-up` starts simplified mode (single shared 6-node cluster) — per CONTEXT.md specifics, this is the default for fast local dev. Sets all `*_REDIS_HOST` env vars to `shared-redis-0` and activates the `simple` profile.
- `dev-cluster` starts full 3-cluster topology (18 Redis nodes) — mirrors production k8s setup. Uses default `*_REDIS_HOST` values (per-domain node-0) and activates the `full` profile.
- `dev-down` stops containers but preserves volumes (data persists across restarts).
- `dev-clean` removes volumes too (fresh start).
- `dev-logs` follows only application service logs (not the Redis node logs which are noisy).
- `test` runs pytest with standalone Redis (unit tests don't need cluster).
